Creating a Text-Based Adventure Game Using Python

February 2022

Something done to have a little fun and stretch my coding skills: a text-based adventure game akin to earlier games such as Zork and Spider and Web. This project is a work-in-progress and will see continued improvements, adding new features and fixing issues as it is developed.

The initial inspiration from this project came from two primary sources: wanting to create a project that demonstrates classes/class inheritance and a coding puzzle from Codewars involving two fighting robots. At this point, I had little to no experience or knowledge of how classes worked in Python. The idea behind this mechanic is that a list of player "classes" would exist that each has its own benefits and cons that can either help or hinder the player depending on what is chosen. Then, that class would be inherited by a basic child "player" class that would ultimately be used throughout the game, simply using the parent class as a starting point. Status of the player is directly modified within the class, such as loss of hit points or increases to attack damage.

One of the strongest features of this particular build of the game is input validation. I was and am aware that modules exist that take care of input validation without much work on the developer, but I wanted to test and see if I could create a validation system that - for the most part - could not be bypassed or broken by the standard player. The downside to such a system is that it takes up lots of space (an entire file is dedicated solely to validation functions) and a different validation system has to be created for each use-case. If a situation calls for a yes or no answer, it is entirely different than one that requires the player to select an item from a list. Efforts have been made to streamline some of these functions, such as ones where players select from multiple options. These have been made more efficient by means of standardizing the function code such that it works with any number of inputs simply by passing a variable to it listing how many options the player has to choose from.

At the moment, the game features the following core features:

  • A functioning main menu
  • Difficuly settings that change the length of the game
  • Player classes that change player stats for each run of the game
  • A semi-randomized environment generator that gives each "room" a unique environment
  • A basic combat system in which players can actively lose health and are given multiple options to choose from
  • A list of enemy classes that are randomly picked from so that encounters can have variety

The next steps for this game are a variety of features that I would personally like to see added such as a player inventory that can provide healing items or weapons to increase attack, further variety to enemies and environments, potential end-of-run bosses that pose a significant challenge. I would also like to implement a save system, allowing players to return to a specific part of their run. This feature, however, poses the biggest challenge and will take some time to implement.

As the game is developed, the end goal is a fully-playable technical demo of a game that can be played and enjoyed by the average person with little to no outside assistance given, other than starting the actual demo. Care is being given to menu options and features such that no prior knowledge is needed to progress and play through the game other than knowing how to type commands. Further improvements to the game will be added as additional updates to the blog.
